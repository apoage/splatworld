# D1 survey — which inverse-rendering implementation to vendor for `decompose`

Planner-run web survey, 2026-07-12 (4 research agents + 1 synthesis judge; evidence from
each repo's README/LICENSE/issue tracker). Supports `tasks/DECISIONS.md` **D1**.

## Judge recommendation

**Vendor: GI-GS (github.com/stopaimme/GI-GS)**

GI-GS is the only candidate whose own code is genuinely MIT and therefore vendorable into the public Apache-2.0 repo — GS-IR, GaussianShader, and R3DG all carry Inria non-commercial licensing at their core or their MIT layer is entangled with NVIDIA-restricted code. Architecturally it is exactly the "GS-IR-style" deferred G-buffer design CLAUDE.md already names, but improved (ICLR 2025) and actively touched as recently as March 2026, versus every rival being abandoned since mid-2024. Its deferred design means all shading and losses run in Python/PyTorch with the rasterizer producing only G-buffers, so the gsplat 1.5.3 port (N-D feature channels + RGB+ED depth) is near-mechanical — matching the "port losses onto gsplat, don't fix legacy kernels" directive — and sm_86/cu124 buildability is the standard low-risk 3DGS stack. Its outputs map one-for-one onto the schema: per-Gaussian albedo (degree-0 RGB), depth-gradient-refined normals, roughness (optional metallic droppable), plus a cubemap env light projectable to SH for the ambient term; and it is the only quality candidate demonstrated on real unbounded vegetation-containing scenes (Mip-NeRF 360 bicycle) rather than synthetic reflective objects.

**Runner-up:** GS-IR (github.com/lzhnb/GS-IR) — same deferred architecture, MIT glue/loss code usable as a donor, identical output attributes; loses on maintenance (dead since July 2024), heavier license contamination (Inria rasterizer fork + NVIDIA renderutils/nvdiffrast woven through the split-sum env light), and no advantage over its own successor.

**Risks / first verification step:** (1) Thin direct evidence: GI-GS has almost no issue-tracker history, so there is no third-party confirmation of a torch 2.x/cu124/py3.11 build of its reference stack, and its pinned env (py3.7, torch 1.12, pytorch3d 0.3.0) will not install as-is — the reference build needs the usual uplift. (2) The screen-space path-traced indirect-illumination pass lives inside the forked Inria rasterizer's forward.cu and cannot be vendored or trivially ported; v1 must drop it or reimplement it as a PyTorch G-buffer ray-march, and albedo quality without it on foliage is unverified. (3) gsplat 1.5.3's N-channel feature rasterization at G-buffer channel counts (~8-9ch + depth) is assumed but not smoke-tested. (4) Decompose quality on the golden synthetic-albedo test and the re-render PSNR budget vs train_base is unproven until run. First verification step: on the local 3090, create a py3.11 + torch cu124 env, build GI-GS's extensions with TORCH_CUDA_ARCH_LIST="8.6" and --no-build-isolation as private (non-committed) scaffolding, run its reference pipeline on one small scene to confirm it trains and produces sane albedo/normal/roughness — in the same session, smoke-test gsplat 1.5.3 rendering an equivalent N-channel G-buffer with gradients to de-risk the port.

**License note:** GI-GS's own repo code is MIT (verified LICENSE, Copyright (c) 2024 stopaimme) — Apache-2.0-compatible; vendor with the MIT notice preserved. Two components must be excluded from the public repo: the diff-gaussian-rasterization submodule fork (Inria/MPII non-commercial Gaussian-Splatting license) — replaced by gsplat (Apache-2.0) — and nvdiffrast (NVIDIA Source Code License, non-commercial) — replace the split-sum env light with a pure-PyTorch or SH environment. Never copy code from the fork's CUDA files (including the indirect pass) into our tree; reimplement from the paper (arXiv 2410.02619) if needed. Record the exclusions and attribution in docs/decisions.md.

## GS-IR — 3D Gaussian Splatting for Inverse Rendering (CVPR 2024)

- **Repo:** https://github.com/lzhnb/GS-IR
- **Last activity:** Code last pushed 2024-07-16; effectively unmaintained since (open issues from late 2024 through Oct 2025 unanswered; repo metadata updated June 2026 is stars/watch noise). 439 stars, 26 forks — no fork has meaningfully modernized it (all 0 stars; a handful of 2025-2026 pushes are personal tweaks).
- **License:** Top-level LICENSE is MIT (oddly, the file is a copy of Mark Kellogg's MIT text; GitHub classifies the repo MIT). BUT the repo vendors restrictively-licensed code: submodules/diff-gaussian-rasterization and simple-knn carry the Inria/MPII "Gaussian-Splatting" license (non-commercial research/evaluation only, not Apache-compatible); pbr/renderutils is copied from NVIDIA nvdiffrec with an NVIDIA all-rights-reserved header ("use...without an express license agreement...strictly prohibited"); required dependency nvdiffrast is under the NVIDIA Source Code License (non-commercial). Even gs-ir/setup.py carries a pasted Inria non-commercial header. Vendoring the whole repo into an Apache-2.0 public repo is NOT clean; only GS-IR's own MIT glue/loss code (train.py, pbr/shade.py logic, gaussian_model.py material params) can be taken, and the Inria rasterizer fork + NVIDIA renderutils/nvdiffrast must be excluded or replaced.
- **CUDA deps:** Four compiled pieces: (1) a FORKED diff-gaussian-rasterization (vendored in-tree, Inria license) extended to splat per-Gaussian normal/albedo/roughness/metallic into G-buffers plus depth/opacity maps with an argmax_depth option and extra backward passes — this coupling to the fork is the core of stage 2; (2) simple-knn (stock Inria); (3) the repo's own "gs-ir" extension (occlusion_kernel.cu + irradiance_kernel.cu) for baking SH occlusion/irradiance volumes — only needed for the optional --indirect path; (4) nvdiffrast + JIT-compiled nvdiffrec renderutils (diffuse_cubemap/specular_cubemap) for the split-sum environment cubemap. Pinned env is ancient: Python 3.7.13, PyTorch 1.12.1, cudatoolkit 11.6 (environment.yml). No arch flags are pinned in any setup.py, so builds follow TORCH_CUDA_ARCH_LIST/detected arch. Known build bugs: gs-ir/src/utils.h uses constexpr uint32_t without #include <cstdint> (breaks GCC 13+/ninja, issue #37; community one-line fix), and gs-ir/setup.py passes a dangling "-I" nvcc flag (copy-paste bug).
- **sm_86 buildability:** No issue in the tracker mentions sm_86/RTX 3090 or cu124 specifically — sm_86 has been supported since CUDA 11.1, so the arch itself is a non-issue; the risk is torch-version and toolchain drift, not the GPU. Direct evidence: issue #15 — a user got the full pipeline running on Ubuntu 22.04 with Python 3.9 + torch 2.1.2 + cu118 after standard fixes; issue #18 — pip build-isolation failures on the rasterizer fork, fixed with `python setup.py develop` / `pip install -e . --no-build-isolation` (one user instead downgraded torch 2.0→1.13, early 2024); issue #37 — ninja failure in the gs-ir extension, fixed by adding #include <cstdint>; issue #38 — same build-isolation "No module named torch" failure. Verdict: buildable on 3090/sm_86 with torch cu124 with high confidence given TORCH_CUDA_ARCH_LIST="8.6", --no-build-isolation, the cstdint patch, and removing the stray "-I" flag — the same rasterizer codebase family builds routinely on torch 2.4/cu124 across the 3DGS ecosystem — but there is no direct cu124/py3.11 report for this exact repo, so budget a small patch session. nvdiffrast itself is actively maintained and builds on cu124.
- **gsplat portability:** High feasibility, and it is the right move for both Blackwell/toolchain and license reasons. The fork's only job is rasterizing extra per-Gaussian channels (normal 3, albedo 3, roughness 1, metallic 1) plus depth/alpha into G-buffers; gsplat 1.5.3's rasterization() takes arbitrary N-D per-Gaussian color channels and render_mode "RGB+ED", covering all of it with gradients, so the G-buffer pass ports directly. The losses are image-space PyTorch and port cleanly: L1+SSIM PBR re-render loss, TV smoothness on material maps, depth-derived-normal vs rendered-normal consistency (recompute pseudo-normals from the gsplat depth map in PyTorch instead of the fork's in-kernel depthToNormal), and envmap regularizers. Two components need replacement rather than porting: (a) the CubemapLight split-sum prefiltering uses NVIDIA renderutils + nvdiffrast (license-restricted) — reimplement diffuse/specular cubemap mips in pure PyTorch or swap in an Apache-clean envlight; (b) the SH occlusion-volume baking (gs-ir CUDA ext) — optional (--indirect flag); skip for v1 middle-distance foliage, revisit later. Estimated port: the deferred-shading loop and losses are a few hundred lines of Python on top of gsplat.
- **Output attributes:** Recovers exactly our target set as first-class per-Gaussian nn.Parameters on the GaussianModel: _albedo (3ch, sigmoid-activated) -> albedo_r/g/b; _normal (3ch, L2-normalized, optimized in stage 2 with depth-derived-normal regularization) -> nx/ny/nz; _roughness (1ch) -> rough; plus _metallic (1ch, optional via --metallic) which our schema drops or folds. These are written into its saved PLY via construct_list_of_attributes, so mapping to splat_relight_schema 1 is a rename in our ply_io.py. Environment light is a single learned global HDR cubemap (6x512x512, exportable to latlong via export_envmap) — a scene-level estimate used to explain away baked lighting, not a per-Gaussian quantity; we discard it at export or keep it for validation renders. No transmission channel — GS-IR assumes opaque microfacets (matches our plan: trans comes from our own stage 5). Albedo lives outside SH (features_dc stays separate), consistent with our SH-degree-0-only rule. Two-stage flow: 30k-iter vanilla-ish training, optional occlusion baking, then 5-10k decomposition iterations — maps well onto our train_base -> decompose split.

**Verdict:** GS-IR is a strong donor for our decompose stage but a poor vendoring target as-is: it recovers precisely the per-Gaussian attributes our schema needs (albedo/normal/roughness, optional metallic) plus a global environment cubemap, its two-stage design maps cleanly onto train_base -> decompose, and its losses are ordinary image-space PyTorch — but the repo is unmaintained (last push July 2024), pinned to Python 3.7/torch 1.12/CUDA 11.6, and its compiled core is a forked Inria-licensed rasterizer plus NVIDIA-restricted envlight code that cannot legally sit in our public Apache-2.0 repo. The pragmatic path, aligned with our "port losses onto gsplat" preference: take the MIT-covered training loop, material parameterization, deferred-shading math, and regularizers; re-host the G-buffer pass on gsplat 1.5.3's N-D channel rasterization + RGB+ED depth (fully supported); reimplement the split-sum cubemap prefilter in pure PyTorch to shed nvdiffrast/renderutils; and skip the optional occlusion-volume baking for v1. If we instead want a quick reference build on the local 3090 (sm_86, cu124, py3.11) to generate ground-truth comparisons, it should compile with TORCH_CUDA_ARCH_LIST="8.6", --no-build-isolation, a one-line cstdint include fix, and removal of a stray "-I" nvcc flag — evidenced by a reported torch 2.1.2/cu118 success, though no one has reported this exact cu124/py3.11 combination, so treat that build as private scaffolding (its licenses also forbid redistributing it in our repo).

## GaussianShader (3D Gaussian Splatting with Shading Functions for Reflective Surfaces, CVPR 2024)

- **Repo:** https://github.com/Asparagus15/GaussianShader
- **Last activity:** Last push 2024-05-20 (effectively unmaintained ~2 years). Issues still being filed through May 2026 (35 total, 22 open); correctness fix PR #20 from April 2024 remains unmerged. 403 stars, 38 forks.
- **License:** NOT Apache-compatible. LICENSE.md is the Inria/GRAPHDECO "gaussian-splatting" license: non-commercial, research/evaluation only, held by Inria + MPII ("THE USER CANNOT USE, EXPLOIT OR DISTRIBUTE THE SOFTWARE FOR COMMERCIAL PURPOSES WITHOUT PRIOR AND EXPLICIT CONSENT"). GitHub's API reports SPDX NOASSERTION. Additionally, scene/NVDIFFREC/ is vendored NVIDIA nvdiffrec code (upstream is NVIDIA Source Code License, non-commercial — I did not find a separate license header in the vendored copy, so this is inferred from upstream). Verbatim vendoring into the public Apache-2.0 splatworld repo would misrepresent licensing and is not permitted; only a reimplementation of the method (losses/shading math onto gsplat, ideas not code) is safe to publish under Apache-2.0. Vendoring privately for non-commercial personal use is within the license terms but the code could not be pushed to the public repo.
- **CUDA deps:** Three CUDA-extension surfaces: (1) submodules/diff-gaussian-rasterization — the VANILLA graphdeco-inria rasterizer (per .gitmodules), used only with colors_precomp; extra rasterization passes render depth, alpha, and normals-as-colors. No custom rasterizer fork. (2) submodules/simple-knn (standard 3DGS densification helper). (3) scene/NVDIFFREC/renderutils — nvdiffrec's JIT CUDA extension (c_src/: bsdf.cu, cubemap.cu, normal.cu, loss.cu, mesh.cu, torch_bindings.cpp) compiled at import via torch.utils.cpp_extension.load(); bsdf/lambert/normal ops have a pure-PyTorch use_python fallback in bsdf.py, but the cubemap prefilter ops (diffuse_cubemap/specular_cubemap) used by the env light are CUDA-only. Plus nvdiffrast==0.3.1 (dr.texture cubemap sampling; the texture op is pure CUDA, no GL context needed). Pinned env is ancient: python 3.7.13, torch 1.10.1+cu111 (environment.yml), and it contains a bogus pip line "cutlass==0.0.1" that is not installable from PyPI (open issue #23) and appears to be an accidental export — the code does not actually import cutlass.
- **sm_86 buildability:** No issue in the tracker reports an sm_86 / RTX 30-series failure; the pinned cu111 toolchain itself already targets sm_86 (cu111 was the first CUDA with Ampere support), so the architecture is not the risk. Reported install problems are environmental, not GPU-arch: #30/#34 conda PackagesNotFoundError and python 3.7 package incompatibilities, #27 missing /usr/local/cuda/bin/nvcc under WSL, #23 the phantom cutlass pip dependency. I found no explicit success report with torch 2.x/cu124 in this repo's issues (uncertainty flag), but every CUDA component it uses — vanilla diff-gaussian-rasterization, simple-knn, nvdiffrast 0.3.1, nvdiffrec renderutils — is routinely built on 3090s with torch 2.x/cu118–cu124 across the 3DGS ecosystem (GS-IR and Relightable3DGaussian ship the same renderutils kernels). Practical path on the 3090 boxes: ignore environment.yml entirely, use Python 3.11 + torch cu124, pip-install the two submodules and nvdiffrast with TORCH_CUDA_ARCH_LIST="8.6", delete the cutlass line. The Python code is 3.7-era but plain; 3.11 should need at most trivial fixes. Expect the usual glm/compiler warnings when building the Inria rasterizer under CUDA 12.4, generally benign — and moot if we port to gsplat.
- **gsplat portability:** High — this is the best property of the repo. The entire shading model is evaluated per-Gaussian in PyTorch (pc.brdf_mlp.shade() on per-Gaussian diffuse/specular/roughness/normal) and the result is passed to the rasterizer as colors_precomp; auxiliary supervision buffers (depth, alpha, normal map, depth-derived reference normal via normal_from_depth_image()) are just extra rasterization passes over per-Gaussian values. That maps directly onto gsplat 1.5.3's N-D feature rasterization / render_mode="RGB+ED" — no custom kernels to port. The losses in train.py are rasterizer-agnostic PyTorch: L1+D-SSIM, predicted_normal_loss(normal, normal_ref, alpha), delta_normal_loss (regularizes the learned normal residual), zero_one_loss on alpha, plus brdf_mlp.clamp_(0,1) on the env light. The one CUDA dependency you must keep or replace is the nvdiffrec environment light: dr.texture cubemap lookups (nvdiffrast) + CUDA-only diffuse_cubemap/specular_cubemap prefilters (renderutils JIT build — small, self-contained, known-good on sm_86); alternatively reimplement the cubemap prefilter in pure torch (it is a one-time-per-step convolution over a 512-cubemap, feasible). One known open bug to patch when porting: get_minimum_axis is wrong (issue #19, unmerged fix PR #20) — directly affects the shortest-axis normal we care about.
- **Output attributes:** Per-Gaussian learnables: diffuse albedo kd (stored in _features_dc; in light.py's shade the diffuse term is dr.texture(diffuse_irradiance_cubemap, normal) * kd, i.e., a true multiplicative albedo-times-irradiance decomposition, so kd is a delit albedo -> albedo_r/g/b, SH0-only as our schema requires); normal = shortest covariance axis + learnable delta normals (_normal/_normal2, saved to PLY as nx,ny,nz and nx2,ny2,nz2) -> nx/ny/nz; roughness (scalar, saved as 'roughness') -> rough; specular tint (_specular) — no slot in our schema, drop or fold at export; residual SH color (_features_rest) — re-bakes view-dependent lighting and MUST be discarded at export per our SH0-only rule. Environment light: trainable nvdiffrec cubemap, 512x512 per face (--brdf_env 512), HDR-exportable (issues #6/#9/#24 discuss .hdr save/load) — satisfies our env-light-estimate requirement. Not provided: transmission (fine — our separate transmission stage handles it) and labels (ours too). Net: covers exactly the albedo/normal/rough/env quartet the decompose stage needs, minus trans/label which we never expected from it.

**Verdict:** Technically a strong donor and a poor vendoring target. The architecture is exactly what we want to port: per-Gaussian shading computed in PyTorch with the rasterizer used as a dumb splatter (colors_precomp + extra feature passes), so the losses and the albedo/normal(shortest-axis+delta)/roughness/512-cubemap-env decomposition transplant cleanly onto gsplat 1.5.3 with no legacy rasterizer kernels — only the small self-contained nvdiffrec cubemap-prefilter CUDA ops (or a torch reimplementation) come along. sm_86/cu124 buildability is low-risk (all components are ecosystem-standard on 3090s), though no direct torch-2.x success report exists in this repo's issues. Two real problems: (1) LICENSE — Inria/GRAPHDECO non-commercial plus vendored NVIDIA nvdiffrec code cannot legally sit in the public Apache-2.0 splatworld repo; we would have to reimplement the losses/shading on gsplat rather than vendor code, which the CLAUDE.md's "prefer porting its losses onto gsplat" stance already accommodates but which contradicts "vendor an existing implementation, don't write the optimization from scratch". (2) Fit — GaussianShader targets shiny/reflective objects; its diffuse decomposition does produce a genuine delit kd (albedo times irradiance-cubemap lookup), but there are open reports of degraded quality on real COLMAP/MipNeRF360 scenes (#10, #14, #25, #35), it is unmaintained with a known unfixed shortest-axis bug (#19/#20), and the residual SH color term can leak baked lighting unless dropped at export. Recommendation: treat it as the reference implementation to port (equations, losses, nvdiffrec env-light module pattern) onto gsplat under our own code, and compare against GS-IR/Relightable3DGaussian before committing — flag the license wall and the vendor-vs-reimplement contradiction to the user before starting M2.

## Relightable 3D Gaussian (R3DG) — "Relightable 3D Gaussian: Real-time Point Cloud Relighting with BRDF Decomposition and Ray Tracing", ECCV 2024 (Gao, Gu et al., NJU/Fudan)

- **Repo:** https://github.com/NJU-3DV/Relightable3DGaussian
- **Last activity:** Last commit 2024-08-05 ("fix datalink in readme.md"), last push 2024-08-14; effectively unmaintained for ~2 years with 52 open issues (678 stars, 54 forks; no meaningful modernized fork found).
- **License:** NOT permissive — root LICENSE.md is the Inria/MPII "Gaussian-Splatting" custom non-commercial research license ("THE USER CANNOT USE, EXPLOIT OR DISTRIBUTE THE SOFTWARE FOR COMMERCIAL PURPOSES WITHOUT PRIOR AND EXPLICIT CONSENT"; research/evaluation only; redistribution only under this same license with the full license text; no sublicensing). The r3dg-rasterization extension is a fork of Inria's diff-gaussian-rasterization and carries the same terms in its headers. This is incompatible with vendoring into a public Apache-2.0 repo: the code cannot be relicensed, and redistribution must stay under the Inria license. Copying its loss code verbatim would carry the license too; a clean-room reimplementation of the paper's losses (paper is arXiv 2311.16043) onto gsplat avoids the problem entirely.
- **CUDA deps:** Three custom CUDA extensions, all in-tree: (1) r3dg-rasterization — a heavily modified diff-gaussian-rasterization fork that additionally splats arbitrary N-channel per-Gaussian features, computes a pseudo-normal from depth in-kernel (computer_pseudo_normal flag), and returns opacity/depth/surface_xyz/num_contrib/weights; (2) bvh ("bvh_tracing") — point-based ray tracer with BVH over Gaussians, used only to bake per-Gaussian SH visibility (gaussian_model.update_visibility); note `from bvh import RayTracer` is a module-level import in scene/gaussian_model.py, so it is a hard dependency even if unused; (3) submodules/simple-knn (standard 3DGS one). Plus pip deps nvdiffrast 0.3.1, torch_scatter 2.1.1, kornia 0.6.12. Pinned stack is very stale: python 3.7.13, torch 1.12.1 + cudatoolkit 11.6 (README recommends compiling extensions with CUDA 11.8). Neither setup.py hardcodes gencode arch — torch auto-detects the GPU, so TORCH_CUDA_ARCH_LIST=8.6 is honored. Importantly, the actual rendering equation (GGX specular, fibonacci-sphere incident sampling, SH eval) and every loss run in pure PyTorch in gaussian_renderer/neilf.py — the CUDA fork is only the feature splatter.
- **sm_86 buildability:** For: the authors' own reference GPU is an RTX 3090 24GB (README: "We run the code in a single NVIDIA GeForce RTX 3090 GPU (24G)"; confirmed again in issue #47) — sm_86 hardware is proven with their pinned stack, and no arch is hardcoded in the setup.py files. Against/uncertain: no issue, fork, or report found of building under torch 2.x / cu12x / python 3.11 — the environment is torch 1.12.1+cu116, py3.7, last touched Aug 2024, and 52 issues sit open unanswered. Issue #3 shows a real build failure of the bvh extension (pybind11 std::uint16_t error — old-torch pybind11 headers vs newer gcc), fixed by users via CUDA 11.8 + env juggling. Expect our torch 2.4/cu124/py3.11 build of the three extensions to need the usual modernization: C++17 flag, missing <cstdint> includes with gcc 12/13, and possible deprecated ATen API touch-ups; no fork was found that has already done this (most-starred fork has 1 star). Memory data point: on a 3090, stage-2 (neilf) OOMs on NeRF-synthetic lego at sample_num=64; users dropped to 40 samples (issue #47) — fine for our small per-asset Gaussian budgets, relevant if scenes grow.
- **gsplat portability:** High — this is the recommended path. The decomposition is: stage 1 vanilla 3DGS (30k it), stage 2 "neilf" (to 50k) where a pure-PyTorch rendering_equation shades each Gaussian (diffuse base_color/pi + GGX specular, fixed fresnel 0.04, incident light = learned global env SH x ray-traced visibility + learned per-Gaussian local incident SH, sampled over ~24-384 fibonacci directions), and the shaded color plus attribute maps are splatted as N-channel features. gsplat 1.5.3 rasterization() natively takes [N, D] per-Gaussian colors with arbitrary D and returns alpha and depth (render_mode RGB+ED), which covers the feature-splatting role; the pseudo-normal-from-depth that r3dg computes in-kernel can be recomputed in PyTorch from the rendered depth map. All losses (L1+DSSIM on both raw and PBR renders, normal-vs-pseudo-normal MSE, optional MVS depth/normal, light-neutrality/whiteness loss, edge-aware smoothness on base_color/roughness, TV on env and normals, mask entropy) are plain tensor ops in calculate_loss — directly portable. The one genuinely non-portable piece is the bvh ray-traced per-Gaussian SH visibility bake; for our middle-distance foliage decompose, v1 can set visibility=1 (or keep it as a free learned SH) and accept less shadow-aware albedo — consistent with our "port losses onto gsplat, don't fix legacy kernels" preference. Caveat: real-scene configs (DTU/TnT) lean on Vis-MVSNet depth/normal supervision; NeRF-synthetic configs train without it, so photogrammetry foliage would run with pseudo-normal supervision only.
- **Output attributes:** Per-Gaussian learned parameters in stage 2: _base_color (3ch diffuse albedo, plain RGB not SH — maps directly to our albedo_r/g/b, satisfying the SH-degree-0-only rule), _normal (explicit per-Gaussian normal vector refined by pseudo-normal/MVS losses — maps to nx/ny/nz), _roughness (1ch [0,1] — maps to rough), plus _incidents_dc/rest (per-Gaussian incident-light SH, 3x16) and _visibility_dc/rest (per-Gaussian visibility SH, 1x16) — these two are baked lighting and must be discarded at export under our schema rules. Global environment light: DirectLightEnv, a single learnable SH environment (scene/derect_light_sh.py; an env-map variant in direct_light_map.py is used for relighting with HDRIs) — gives us the environment-light estimate the decompose stage needs. No metallic, and no transmission — trans stays our own downstream stage as planned. R3DG's inverse rendering assumes opaque GGX surfaces, so the known thin-leaf weakness applies exactly as CLAUDE.md anticipates.

**Verdict:** R3DG is an excellent algorithmic match for our decompose stage — it recovers exactly our schema's albedo/normal/roughness per Gaussian plus a global SH environment-light estimate, its authors developed it on the very GPU class we target (RTX 3090, sm_86), and critically its entire BRDF/lighting optimization and every loss live in pure PyTorch, with the custom CUDA fork serving only as an N-channel feature splatter that gsplat 1.5.3 can replace almost one-for-one (drop the BVH visibility bake for v1). But it cannot be vendored: the whole repo, including the rasterizer fork, is under the Inria non-commercial Gaussian-Splatting license, which forbids redistribution into our public Apache-2.0 repo and forbids relicensing; it is also unmaintained since Aug 2024 on a torch 1.12/py3.7 stack with no confirmed torch 2.x/cu124 build reports. Recommended fit: use R3DG as the reference design and reimplement its stage-2 losses and rendering equation onto gsplat (clean-room from the paper, arXiv 2311.16043, to stay license-safe) rather than building its legacy extensions — which is also our stated technical preference; if we ever want its code verbatim, it must stay out of the published repo. Uncertainty flagged: gsplat's exact per-channel feature limits at D≈20 channels should be smoke-tested, and no third party has publicly confirmed a cu124 build of the extensions either way.

## GI-GS (Global Illumination Decomposition on Gaussian Splatting for Inverse Rendering, ICLR 2025)

- **Repo:** https://github.com/stopaimme/GI-GS
- **Last activity:** Initial commit Oct 4, 2024; code release Jan 2025; latest commit Mar 12, 2026 ("Update albedo processing and metrics calculation") — actively touched within the last 4 months. 131 stars, 27 commits.
- **License:** MIT for the repo's own code (LICENSE verified: "MIT License, Copyright (c) 2024 stopaimme"; no Inria/non-commercial clauses). Caveat: its vendored submodules include a fork of Inria's diff-gaussian-rasterization, which carries the upstream Gaussian-Splatting research/non-commercial license — that submodule must NOT be vendored into the Apache-2.0 repo. The MIT Python layer (losses, PBR shading, training loop) is safe to vendor; replacing the rasterizer with gsplat (Apache-2.0) eliminates the tainted component entirely.
- **CUDA deps:** Three CUDA extensions: (1) custom fork of Inria diff-gaussian-rasterization extended with G-buffer outputs (depth/normal/albedo/roughness/metallic) plus a screen-space "lightweight path tracing" indirect-illumination pass implemented inside cuda_rasterizer/forward.cu (~lines 635-910) — this is the only deep coupling to the base rasterizer; (2) nvdiffrast (cubemap/split-sum environment light); (3) simple-knn. Pinned env is legacy, inherited from GS-IR: python 3.7.13, torch 1.12.1, cudatoolkit 11.6, pytorch3d 0.3.0 — it will not install as-pinned on a cu124 stack and needs an env uplift (torch 2.x/cu12x), which the community routinely does for Inria-fork rasterizers.
- **sm_86 buildability:** No architecture-specific problems expected or reported: sm_86 is squarely inside what the Inria rasterizer fork, nvdiffrast, and simple-knn support (these are the exact kernels the 3DGS community builds on 3090s daily; the wheel/arch problem in CLAUDE.md is a Blackwell sm_120 issue, not sm_86). Issue tracker shows no build-failure or GPU-compat reports — only one open issue (checkpoint loading, Mar 2026). Direct positive evidence is thin because the repo has few issues, but the components are the standard stack: expect the usual torch-2.x/cu12x uplift work (glm header include fix for diff-gaussian-rasterization, drop the pytorch3d 0.3.0 pin) rather than any sm_86 kernel problem. Papers-adjacent confirmation: SSD-GS (ICLR 2026) builds this same class of extensions on torch 2.4.1 + CUDA 12.4.
- **gsplat portability:** Good — the best of any surveyed candidate, because GI-GS is deferred-shading by design: the rasterizer only produces G-buffers (albedo/normal/roughness/metallic/depth), and all PBR shading, env-light evaluation, and losses run in Python/PyTorch. gsplat 1.5.3's rasterization() already renders N-channel per-Gaussian features and depth (RGB+ED); normals render as feature channels. So the material/geometry losses port near-mechanically, matching CLAUDE.md's "port losses onto gsplat" directive. The one non-portable piece is the CUDA screen-space path-traced indirect illumination (forward.cu 635-910): either reimplement as a PyTorch screen-space ray-march over the G-buffer (it consumes only depth/normal/prior-render — no per-Gaussian traversal), or drop indirect lighting for v1, which is defensible for middle-distance foliage where the runtime ambient term covers it. nvdiffrast split-sum env light can be kept (builds fine on sm_86) or swapped for a pure-torch SH environment.
- **Output attributes:** Per-Gaussian: albedo (RGB, degree-0 — maps to albedo_r/g/b), roughness (maps to rough), optional metallic (drop or fold into rough), normals refined via depth-gradient pseudo-normal supervision (maps to nx/ny/nz — exactly the "decompose refines shortest-axis normals" role). Global: environment light as a cubemap (split-sum) — project to SH for the runtime ambient_sh(N) term and for validating delighting. Does not produce trans or label — fine, those are our separate transmission/label stages. Verified relight/eval scripts (relight.py, relight_eval.py, normal_eval.py) plus albedo metrics ease building our PSNR-budget metrics.json gates. Proven on real unbounded Mip-NeRF 360 scenes (bicycle — vegetation-heavy) as well as TensoIR synthetic, unlike the object-centric alternatives.

**Verdict:** GI-GS is the single best newer candidate for the decompose stage: it is literally a strictly-improved successor to GS-IR (same deferred G-buffer architecture CLAUDE.md already names as the model — "GS-IR-style" — but with better indirect illumination and albedo quality, ICLR 2025), its own code is MIT and cleanly vendorable into the Apache-2.0 repo, it recovers exactly the per-Gaussian albedo/normal/roughness + env-light set our schema needs, it is the only quality 2024-26 IR method demonstrated on real unbounded vegetation-containing scenes (Mip-NeRF 360) rather than synthetic objects, it saw maintenance commits as recently as March 2026, and its deferred design makes the gsplat-1.5.3 port mostly a Python-side exercise with one droppable CUDA piece (screen-space indirect). Runners-up rejected: GeoSplatting (Apache-2.0 and gsplat-native, but FlexiCubes mesh guidance is fundamentally wrong for thin foliage and it targets synthetic objects), IRGS/Ref-Gaussian/GS-2DGS (MIT but reflective-object-centric with a custom CMake 2D-Gaussian ray tracer that cannot be ported to gsplat), SSD-GS (matches our cu124 env and borrows gsplat, but GPL-3.0 and requires point-light/OLAT-style captures — incompatible with fixed-illumination photogrammetry), LumiGauss (outdoor in-the-wild fit but no roughness channel and 2DGS non-commercial rasterizer), ROS-GS/R3GW (relevant papers, no public code). Recommended path: vendor GI-GS's MIT Python layer, rebuild its G-buffer pass on gsplat 1.5.3, drop or torch-reimplement the indirect pass, and validate against the golden synthetic-albedo test before trusting it on foliage.
